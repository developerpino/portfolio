01
하네스 — 지금 만들고 있는 것

  
카페24 Agent는 상점 데이터를 보고 스스로 움직이는 자율 운영 루프입니다. 기성 프레임워크의 조립이 아니라 순환 구조와 통제 장치를 바닥부터 설계했습니다. 공고의 하네스 구성 요소와 하나씩 맞물립니다.

  

    
    

      
orchestration수집→분석→도출→실행→메모리 순환 루프를 직접 설계. 액션마다 자율 실행과 승인 필요를 구분하는 권한 체계 포함

      
memory실행 결과와 맥락을 축적해 다음 분석에 재투입 — 루프가 돌수록 판단이 정교해지는 구조

      
tool 연동쇼핑몰 솔루션 API를 입출력이 표준화된 액션 단위로 도구화. AI는 검증된 액션의 조합만 실행

      
평가·관측실행 이력 추적으로 실패 구간을 찾아 고치는 개선 사이클 — 하루 500만 회 실행 규모에서 수개월 운영

      
안전 게이트실물에 닿는 액션은 Server Driven UI 승인 화면을 거쳐야 실행. 판단과 실행의 분리를 구조에 내장

    

  






  
02
연혁 — 도메인은 바뀌어도 하는 일은 같았습니다

  
게임, 물류, 글로벌 커머스, AI. 현장의 반복 업무와 판단을 데이터로 구조화하고 시스템이 대신 실행하게 만드는 일을 16년째 하고 있습니다.

  

    
2010
윈디소프트 — 첫 직장, 게임 웹 풀스택
겟앰프드2·괴혼온라인·러스티하츠 온라인의 런처 연동 공식 사이트를 개발했습니다. PHP 비즈니스 로직과 Java 인증, 4종 DB 운용, IE6 대응까지 — 제약 많은 환경에서 기본기를 다졌습니다.

    
2011

2018

2026

문제
기존 AI는 사람이 물어봐야 움직였습니다. 다음 단계는 사람이 지시하거나 플로우를 만들 필요조차 없는 자율 운영인데, 자율성이 커질수록 통제가 서비스의 생명이 됩니다.

    • 
        
  • 액션마다 자율 실행과 승인 필요를 구분하는 권한 설계, 산출물 품질을 검증하는 평가 단계를 루프 안에 배치

  • 사내 시스템과 AI 모델 간 인터페이스 체계 정립까지 실서비스 배포용 아키텍처 전반 담당

    P-02개발 하네스사내 운영2026 ~▸
      "구현을 잘하는 Agent"보다 "질문을 잘하는 Agent" — 이슈에서 PR까지 품질 게이트를 거쳐 도는 루프
    
    

      
문제
코딩 Agent는 도구만 던져 주는 방식으로는 효과가 나지 않습니다. 실무 이슈는 재현 조건, 기대 결과, 테스트 기준이 빠진 채 들어오는 경우가 대부분이라, AI가 바로 구현에 들어가면 빠른 것 같아도 엉뚱한 방향으로 갑니다.

      
설계
    
        
  • Jira/Slack 이슈 확인 → 부족한 요구사항을 Agent가 먼저 질문으로 보완 → 작업 가능 상태 판단
    • 
        
  • 구현 → 테스트 → E2E 검수를 통과해야만 GitHub PR 생성 — 통과 못 한 산출물은 프로세스가 걸러냄
    • 
        
  • 잘못된 전제로 작업하는 것을 구조적으로 차단, AI 산출물의 신뢰성을 사람의 주의력이 아니라 프로세스로 보장
    • 
      

      
의미
공고의 "하네스 위에 개발자가 subagent·skill·workflow를 올리는 인터페이스"와 "평가 체계·실패 구간 분석" — 그 두 가지를 개발 조직 규모에서 미리 연습해 본 경험입니다.

      
Claude CodeOpenAI CodexJira/Slack 연동E2E 검수GitHub PR 자동화

    

  


  

    P-03AI Studio운영 중2025▸
      누구나 자연어로 업무 플로우를 만들어 올리는 플랫폼 — 5만여 개 상점, 하루 500만 회 실행
    
    

      
문제
운영대행 업무는 반복적이지만 상점마다 조건과 예외가 달라서 개발자가 매번 자동화를 만들어 주는 방식으로는 확장이 안 됐습니다. 가장 어려운 지점은 모호한 자연어를 신뢰할 수 있는 실행 단위로 바꾸는 일이었습니다. LLM은 같은 요청에도 다른 답을 내놓았고, 잘못된 실행 한 번이 실제 주문과 정산을 건드릴 수 있었습니다.

      
설계
    
        
  • 업무를 액션 단위로 분해하고 입출력을 표준화 — AI는 검증된 액션의 조합만 구성
    • 
        
  • 플로우 실행·스케줄링·공유와 실제 API 액션 연동, 실행 이력 추적 구조까지 포함
    • 
        
  • 실행 로그에서 실패 패턴을 찾아 고치는 운영 안정화를 수개월 반복 — 될 때까지 매달리는 것 말고 다른 방법이 없었습니다
    • 
      

      
성과
운영대행 계약 상점 5만여 개의 수백 가지 업무가 하루 500만 회 넘게 자동 실행됩니다. 수작업으로 상점을 하나씩 처리하던 운영자들은 자연어로 플로우를 설계하는 기획자로 바뀌었습니다. 실험이 아니라 실서비스 트래픽을 감당하는 운영 단계까지 완주한 프로젝트입니다.

    •

04
설계 원칙 — 실서비스로 검증한 네 가지

  
3년간 얻은 결론은 하나입니다. LLM 서비스의 성패는 모델 성능이 아니라 실행을 통제하는 설계에 달려 있습니다. 의료는 이 원칙들이 가장 무겁게 적용되어야 하는 도메인이라고 생각합니다.

  

    
PRINCIPLE 1
판단과 실행의 분리
AI 상점 비서 · 2024
AI가 판단 즉시 실행하게 두지 않습니다. 구조화 출력 → 동적 승인 UI → 사람 승인 후 실행 → 전 단계 로그. 임의 실행이라는 가장 큰 두려움을 눈에 보이는 구조로 해소합니다.

    
PRINCIPLE 2
액션 단위 표준화
AI Studio · 2025
모호한 자연어를 신뢰할 수 있는 실행으로 바꾸는 방법은 AI에게 자유가 아니라 검증된 선택지를 주는 것이었습니다. 자유도와 신뢰성은 액션 표준화라는 층을 사이에 둘 때 양립했습니다.

    
PRINCIPLE 3
품질 게이트 내장
개발 하네스 · 2026
AI 산출물의 품질을 사람의 사후 검토에 맡기지 않습니다. 요건이 부족하면 질문이 먼저 나가고, 검수를 통과해야만 다음 단계가 열립니다. 신뢰성은 프로세스가 보장합니다.

    
PRINCIPLE 4
순환과 평가
카페24 Agent · 2026~
자율 Agent는 실행하고 끝나지 않고 결과를 다시 배워야 합니다. 평가 단계를 루프 안에 배치해 루프가 돌수록 판단이 정교해지되, 품질이 떨어지면 루프가 스스로 감지합니다.
